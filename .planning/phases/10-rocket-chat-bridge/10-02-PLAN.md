**1. Create `crates/bm/tests/e2e/rocketchat.rs` — RcPodGuard cleanup helper:**

This is fundamentally different from TgMock. TgMock simulates an *external* server that exists independently of `bm`. For RC, `bm bridge start` IS what creates the pod (RC is a *local* bridge whose lifecycle is managed by the Justfile). So the test helper is NOT responsible for starting the pod — only for cleanup if the test panics before `bm bridge stop` runs.

Create an `RcPodGuard` struct:

```rust
pub struct RcPodGuard {
    pod_name: String,
    port: u16,
}
```

Methods:
- `RcPodGuard::new(pod_name: String, port: u16) -> Self` — Wraps an already-running pod (created by `bm bridge start`). The test creates this guard AFTER `bm bridge start` succeeds, reading pod_name from bridge-state.json.
- `RcPodGuard::port(&self) -> u16`
- `RcPodGuard::pod_name(&self) -> &str`
- `RcPodGuard::from_existing(pod_name: String, port: u16) -> Self` — For progressive mode (alias for `new`)
- `RcPodGuard::into_parts(self) -> (String, u16)` — For progressive mode, consumes self WITHOUT running Drop
- `impl Drop for RcPodGuard` — Safety net: `podman pod rm -f {pod_name}` (force-remove handles both running and stopped pods). This only fires if the test panics between `bm bridge start` and `bm bridge stop`.

The guard does NOT have: `start()`, `admin_token`, `admin_user_id`, `create_token_secret`, `is_healthy()`, or any health check methods. All of that is handled by the Justfile recipes invoked through `bm` commands.
