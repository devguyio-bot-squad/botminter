  <action>
Create `.planning/specs/bridge/bridge-spec.md` -- the primary bridge plugin specification.

**Structure** (follow Knative-style spec organization):

1. **Title and Status**: "Bridge Plugin Specification v1alpha1", Status: Draft

2. **Conformance**: Standard RFC 2119 boilerplate paragraph. Keywords MUST be UPPERCASE.

3. **Overview**: Brief description of what a bridge is (a pluggable communication integration for BotMinter teams), what the spec covers, and who it's for (bridge implementors).

4. **Terminology**: Define key terms -- bridge, local bridge, external bridge, bridge manifest, bridge schema, config exchange, identity, lifecycle.

5. **Bridge Types**:
   - **Local bridge**: Full lifecycle management. BotMinter starts/stops the service. Examples: Rocket.Chat (self-hosted). MUST implement lifecycle AND identity commands.
   - **External bridge**: Identity management only. Service runs independently. Examples: Telegram (SaaS). MUST implement identity commands. MUST NOT declare lifecycle commands.

6. **Bridge Manifest (`bridge.yml`)**:
   - MUST be at bridge root directory
   - Required fields with types and constraints:
     - `apiVersion`: MUST be `botminter.dev/v1alpha1`
     - `kind`: MUST be `Bridge`
     - `metadata.name`: MUST be non-empty string matching `[a-z][a-z0-9-]*`
     - `metadata.displayName`: SHOULD be present, human-readable name
     - `metadata.description`: SHOULD be present
     - `spec.type`: MUST be `local` or `external`
     - `spec.configSchema`: MUST reference a valid JSON Schema file relative to bridge root
     - `spec.lifecycle` (local only): MUST contain `start`, `stop`, `health` -- each a Justfile recipe name
     - `spec.identity`: MUST contain `onboard`, `rotate-credentials`, `remove` -- each a Justfile recipe name
     - `spec.configDir`: MUST specify config exchange directory, typically `$BRIDGE_CONFIG_DIR`
   - Include inline YAML snippet for both local and external types (brief, point to examples/ for complete versions)

7. **Config Schema (`schema.json`)**:
   - MUST be a valid JSON Schema (Draft 2020-12 RECOMMENDED)
   - MUST have `type: "object"` at root
   - MUST define `properties` for bridge-specific configuration
   - SHOULD define `required` array for mandatory config values
   - BotMinter validates config against schema before invoking any bridge command
   - Schema is bridge-specific -- each bridge defines its own config shape

8. **Lifecycle Commands** (local bridges only):
   - `start`: MUST start the bridge service. MUST write service config to `$BRIDGE_CONFIG_DIR/config.json` on success.
   - `stop`: MUST stop the bridge service. SHOULD clean up resources.
   - `health`: MUST exit 0 if service is healthy, non-zero otherwise. SHOULD write health status to stderr.
   - All commands are Justfile recipes. No hardcoded command names in the contract.

9. **Identity Commands** (all bridges):
   - `onboard <username>`: MUST create a user/bot on the bridge. MUST write credentials to `$BRIDGE_CONFIG_DIR/config.json`.
   - `rotate-credentials <username>`: MUST generate new credentials for existing user. MUST write new credentials to `$BRIDGE_CONFIG_DIR/config.json`.
   - `remove <username>`: MUST remove the user from the bridge. SHOULD clean up associated resources.
   - Username is passed as a Justfile recipe argument.

10. **Config Exchange**:
    - All bridge commands that produce configuration MUST write to `$BRIDGE_CONFIG_DIR/config.json` (not stdout)
    - `$BRIDGE_CONFIG_DIR` is set by BotMinter before invoking any command
    - Output format: JSON object with command-specific fields
    - Rationale: file-based exchange avoids stdout corruption from diagnostic output
    - Define expected JSON shapes for each command output (start, onboard, rotate-credentials)

11. **Environment Variables**:
    - `$BRIDGE_CONFIG_DIR`: Set by BotMinter. Commands write output here.
    - `$BM_TEAM_NAME`: Set by BotMinter. Team context for multi-team setups.
    - Bridge-specific env vars: defined in the bridge's `schema.json` and resolved by BotMinter.

12. **Directory Structure**:
    - A conformant bridge implementation MUST contain: `bridge.yml`, `schema.json`, `Justfile`
    - Layout: bridge root directory with these 3 files. Additional support files are permitted.

13. **Non-Goals** (per CONTEXT.md decision):
    - This spec does NOT define: command invocation mechanisms (how BotMinter calls recipes), error handling policies, retry/backoff logic, health check intervals, runtime state management, multi-bridge coordination, bridge discovery/registration. Those are implementation concerns for Phase 8+.

14. **Conformance**:
    - A bridge is conformant if: `bridge.yml` parses as valid YAML with all required fields, `schema.json` parses as valid JSON Schema, `Justfile` contains recipes matching all declared commands, and the bridge type correctly determines which command categories are present.
    - Conformance is structural -- runtime behavior testing is separate.

15. **References**: Link to examples/ directory, ADR-0002 (bridge abstraction), RFC 2119.

Use RFC 2119 keywords (MUST, SHOULD, MAY) consistently. The spec should feel like a Knative resource spec -- declarative, versioned, with clear conformance levels.

Create complete, working reference examples in `.planning/specs/bridge/examples/`.

1. **`bridge.yml`** -- Local bridge reference (Rocket.Chat-style):
```yaml
apiVersion: botminter.dev/v1alpha1
kind: Bridge
metadata:
  name: rocketchat
  displayName: "Rocket.Chat"
  description: "Self-hosted team chat via Rocket.Chat and MongoDB"

spec:
  type: local
  configSchema: schema.json

  lifecycle:
    start: start
    stop: stop
    health: health

  identity:
    onboard: onboard
    rotate-credentials: rotate
    remove: remove

  configDir: "$BRIDGE_CONFIG_DIR"
```

2. **`bridge-external.yml`** -- External bridge reference (Telegram-style):
```yaml
apiVersion: botminter.dev/v1alpha1
kind: Bridge
metadata:
  name: telegram
  displayName: "Telegram"
  description: "Telegram bot integration (external service)"

spec:
  type: external
  configSchema: schema.json

  identity:
    onboard: onboard
    rotate-credentials: rotate
    remove: remove

  configDir: "$BRIDGE_CONFIG_DIR"
```
Note: NO `lifecycle` section for external type.

3. **`schema.json`** -- Reference config schema (Rocket.Chat-style):
```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "title": "Rocket.Chat Bridge Configuration",
  "description": "Configuration for the Rocket.Chat bridge",
  "type": "object",
  "properties": {
    "host": {
      "type": "string",
      "description": "Rocket.Chat server hostname",
      "default": "localhost"
    },
    "port": {
      "type": "integer",
      "description": "Rocket.Chat server port",
      "default": 3000
    },
    "admin_username": {
      "type": "string",
      "description": "Admin username for API operations"
    },
    "admin_password": {
      "type": "string",
      "description": "Admin password for API operations"
    },
    "mongodb_port": {
      "type": "integer",
      "description": "MongoDB port",
      "default": 27017
    }
  },
  "required": ["admin_username", "admin_password"]
}
```

These are illustrative reference examples, not implementations. They show what a conformant bridge's files look like. The actual Rocket.Chat and Telegram bridges will be built in Phases 9 and 11.
