   - Status: accepted, Date: 2026-03-08
   - Context: BotMinter needs pluggable communication bridges (Telegram, Rocket.Chat, future services)
   - Decision Drivers: operator self-hosts, no vendor lock-in, existing Justfile recipe pattern, bridge authors need clear contract
   - Considered Options: (a) Rust trait plugin system, (b) Shell script bridge with YAML manifest, (c) gRPC bridge protocol, (d) REST API bridge protocol
   - Decision: Shell script bridge with YAML manifest (`bridge.yml` + `schema.json` + Justfile recipes)
   - Key decisions to document: local vs external bridge types, file-based config exchange (`$BRIDGE_CONFIG_DIR/config.json` not stdout), identity commands (onboard/rotate-credentials/remove), lifecycle commands (start/stop/health for local only), apiVersion `botminter.dev/v1alpha1`

5. **`0003-ralph-robot-backend.md`** -- Ralph robot backend decisions.
   - Status: accepted, Date: 2026-03-08
   - Context: Ralph Orchestrator already has pluggable robot backends (RobotConfig, RobotService trait). BotMinter bridges must map to Ralph's robot abstraction.
   - Decision Drivers: Ralph already pluggable, bridge credentials become robot credentials, minimal coupling between bridge and robot layers
   - Considered Options: (a) Bridge generates ralph.yml robot section directly, (b) Bridge outputs credentials and BotMinter maps to ralph.yml, (c) Ralph reads bridge config directly
   - Decision: Bridge outputs credentials via file-based exchange, BotMinter maps them to ralph.yml robot section during `bm teams sync`
   - Consequences: clean separation (bridge knows nothing about Ralph), BotMinter is the translation layer
   - Confirmation: `bm teams sync` generates valid ralph.yml with robot section from bridge credentials (Phase 10)

All ADRs use MADR 4.0.0 format per the template. Keep each ADR to 1-2 pages. ADRs document the "why" not the "what" -- the bridge spec (Plan 02) covers the contract details.
