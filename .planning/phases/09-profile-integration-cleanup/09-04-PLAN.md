  <files>docs/content/concepts/bridges.md, docs/content/how-to/bridge-setup.md, docs/mkdocs.yml</files>
  <action>
1. **Create `docs/content/concepts/bridges.md`** -- Bridge concepts page covering:
   - What bridges are (pluggable communication integrations)
   - Two bridge types: local (managed, full lifecycle) vs external (SaaS, identity-only)
   - Per-member identity model (each member gets own bot user/token)
   - Credential flow: bridge onboard -> config exchange -> system keyring (local formation) -> env var injection at launch
   - Formation-aware credential storage: system keyring for local formations, K8s Secrets planned for K8s formations
   - Key principle: secrets live in the keyring, never in bridge-state.json or ralph.yml; injected as env vars at runtime
   - Profile bridge declaration (botminter.yml `bridges:` field)
   - Bridge spec overview with link to the spec document
   - **Headless/CI environments**: document `BM_BRIDGE_TOKEN_{USERNAME}` env var as the official approach when system keyring is unavailable. This is the primary credential mechanism for CI pipelines and containers.
   - **Security considerations**: credentials stored in system keyring (never in bridge-state.json, ralph.yml, or config.yml); env vars visible to the Ralph process and its children; keyring entries tied to the machine (re-provision after migration)

2. **Create `docs/content/how-to/bridge-setup.md`** -- Step-by-step guide:
   - Prerequisites: profile with bridge support (e.g., scrum-compact with Telegram)
   - Step 1: `bm init` -- select bridge during wizard (or `--bridge telegram` for non-interactive)
   - Step 2: `bm hire` -- optionally provide per-member bot tokens during hire
   - Step 3: `bm teams sync --bridge` -- provisions identities and rooms
   - Step 4: `bm start` -- launches members with bridge credentials injected
   - Adding credentials after hire: `bm bridge identity add <username>`
   - Checking bridge status: `bm bridge status`
   - External bridge specifics (Telegram): create bots via @BotFather, supply tokens during hire

3. **Update `docs/mkdocs.yml`** -- Add navigation entries:

   - Conformance checklist includes room section validation
   - If the spec is missing anything else that was implemented in Phase 8, add it

Do NOT create documentation files for deferred features (K8s secrets, multi-bridge, etc.). Only document what exists now.
