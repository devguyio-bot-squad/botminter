**Bridge provisioning (commands/teams.rs):**

1. Implement `provision_bridge()` function called when `--bridge` or `--all` flag is set:
```rust
fn provision_bridge(
    team_repo: &Path,
    team_name: &str,
    workzone: &Path,
    members: &[String],
) -> Result<()>
```

2. Inside `provision_bridge()`:
   a. Call `bridge::discover(team_repo, team_name)` to find bridge dir. If None, print "No bridge configured -- skipping" and return Ok.
   b. Load bridge manifest via `bridge::load_manifest()`.
   c. Load bridge state via `bridge::load_state()`.
   d. Create `LocalCredentialStore::new(team_name, bridge_name, state_path)` — tokens go to system keyring.
   e. For each member NOT already in state.identities:
      - If bridge type is "external": check if credential exists via `credential_store.retrieve(member)`. If None, print warning "  {member}: no bridge credentials -- skipping. Use `bm bridge identity add` to add later." and continue.
      - If bridge type is "external" AND credential exists: set `BM_BRIDGE_TOKEN_{USERNAME}` env var before invoking onboard recipe (the recipe validates and registers the token).
      - If bridge type is "local" (managed): invoke onboard recipe directly (it creates the user and returns credentials).
      - Call `bridge::invoke_recipe()` with the identity.onboard recipe name and member as arg.
      - If recipe returns config JSON with username/user_id/token: store identity metadata (username, user_id, created_at) in bridge-state.json AND store the token in credential store (system keyring). The token does NOT go into bridge-state.json.
   f. Create team room if `state.rooms.is_empty()` and manifest has room spec.
   g. Save bridge state.

3. Wire `provision_bridge()` into the sync flow: call it after workspace assembly when `bridge_flag` is true.

NOTE: `inject_robot_enabled()` runs during ALL sync operations (called from `sync_workspace()`), not just when `--bridge` is set. This ensures that `bm teams sync` (without --bridge) after `bm bridge identity add` correctly updates RObot.enabled. The `--bridge` flag only controls whether the provisioning step (onboard recipes, room creation) runs.

**ralph.yml RObot injection (workspace.rs):**

1. Add a function `inject_robot_enabled()` that runs during `sync_workspace()`:
```rust
pub fn inject_robot_enabled(
    ralph_yml_path: &Path,
    member_has_credentials: bool,
) -> Result<()>
```

2. This function:
   - Loads ralph.yml as `serde_yml::Value`
   - Sets `doc["RObot"]["enabled"]` to `true` if member_has_credentials, `false` otherwise
   - Does NOT write any token, secret, or credential to ralph.yml
   - Preserves all other ralph.yml content (timeout_seconds, checkin_interval_seconds, etc.)
   - Writes back to disk

3. Call `inject_robot_enabled()` from the sync workflow. To determine if member has credentials:
   - If the team has NO bridge configured (bridge::discover() returns None): skip `inject_robot_enabled()` entirely — leave ralph.yml untouched. No bridge means no RObot section to manage.
   - If the team HAS a bridge configured: use `credential_store.retrieve(member_name)?.is_some()` — checks system keyring
   - Do NOT rely on bridge-state.json identity presence alone (identity metadata may exist without a token if the operator hired without providing one)

CRITICAL per ADR-0003: ralph.yml only gets RObot.enabled = true/false. NO secrets. Secrets are injected as env vars by `bm start` (Task 2).

1. Modify `bm start` to resolve per-member credentials instead of using the team-wide `telegram_bot_token`:

**Current code (line ~109):**
```rust
let telegram_token = team.credentials.telegram_bot_token.as_deref();
```
This passes the SAME token to ALL members. Replace with per-member resolution.

**New approach:**
- Before the member launch loop, discover the bridge and set up the credential store:
  ```rust
  // Discover bridge from team repo
  let team_repo = cfg.workzone.join(&team.name).join("team");
  let bridge_dir = bridge::discover(&team_repo, &team.name)?;
  let (credential_store, has_bridge) = if let Some(ref dir) = bridge_dir {
      let manifest = bridge::load_manifest(dir)?;
      let state_path = bridge::state_path(&cfg.workzone, &team.name);
      let store = bridge::LocalCredentialStore::new(
          &team.name, &manifest.metadata.name, state_path,
      );
      (Some(store), true)
  } else {
      (None, false)
  };
  ```
- Inside the per-member loop, resolve the credential for THIS member:
  ```rust
  // Resolve per-member bridge credential: env var first, then keyring
  let member_token = if let Some(ref store) = credential_store {
      bridge::resolve_credential_from_store(member_dir_name, store)?
  } else {
      None
  };
  ```
- Pass `member_token.as_deref()` to `launch_ralph()` instead of the team-wide token.
- Remove the team-wide `team.credentials.telegram_bot_token` usage entirely — per-member resolution replaces it.

2. Update `launch_ralph()` to accept the credential as before (it already takes `Option<&str>`). The env var name (`RALPH_TELEGRAM_BOT_TOKEN`) stays the same -- Ralph expects this name.

3. For future bridge types (Rocket.Chat), the env var name would differ. For now, Telegram is the only bridge, so `RALPH_TELEGRAM_BOT_TOKEN` is correct. Add a TODO comment noting this should become bridge-type-aware when more bridges are added.

4. **Diagnostic warning for forgotten sync --bridge**: After resolving per-member credentials, if a member has a credential in the store but `ralph.yml` has `RObot.enabled = false`, print a warning:
   `"Warning: {member} has bridge credentials but RObot is disabled in ralph.yml. Run 'bm teams sync' to update."`
   This catches the common case where the operator hired with a token or used `bm bridge identity add` but forgot to run `bm teams sync` afterward.

CRITICAL: Each Ralph process gets its OWN credential via `resolve_credential_from_store` (env var `BM_BRIDGE_TOKEN_{USERNAME}` with hyphens→underscores first, then system keyring). No team-wide token. No bridge-state.json fallback for tokens.
