<task type="auto" tdd="true">
  <name>Task 1: Init wizard bridge selection and hire token prompt</name>
  <files>crates/bm/src/commands/init.rs, crates/bm/src/commands/hire.rs</files>
  <behavior>
    - Test: Non-interactive init with --bridge telegram records "bridge: telegram" in team botminter.yml
    - Test: Non-interactive init without --bridge produces no bridge key in team botminter.yml
    - Test: Non-interactive init with --bridge for a bridge not in the profile fails with error
    - Test: Hire with external bridge configured stores token when provided (integration test with stub)
  </behavior>
  <action>
**Init wizard (commands/init.rs):**

1. After profile selection (both interactive and non-interactive paths), read the ProfileManifest to get available bridges.

2. **Interactive mode:** If `manifest.bridges` is not empty, add a `cliclack::select` step:
   - If `--bridge` flag was provided on the command line, use it as the pre-selected value (skip the interactive prompt). Validate it exists in the profile's bridges list — if not found, bail with error listing available bridges.
   - Otherwise, present the `cliclack::select` prompt:
     - List each bridge with display_name and description
     - Add a "No bridge" option (value "none")
     - If user selects a bridge (not "none"), record it
   - If profile has no bridges, skip this step entirely (regardless of `--bridge` flag — if `--bridge` is passed for a profile with no bridges, bail with error)

3. **Non-interactive mode:** If `--bridge` flag is provided:
   - Validate the bridge name exists in the profile's bridges list
   - If not found, bail with error listing available bridges
   - If `--bridge` is omitted, no bridge is configured (this is fine)

4. **Recording bridge selection:** After team repo is created and botminter.yml is extracted, if a bridge was selected:
   - Load the team's botminter.yml as serde_yml::Value
   - Set `doc["bridge"] = serde_yml::Value::String(bridge_name)`
   - Write back to the team's botminter.yml
   - This is the field that `bridge::discover()` already reads

5. Init does NOT start the bridge, does NOT prompt for tokens. It only records the selection.

**Hire flow (commands/hire.rs):**

1. After the member directory is created and committed, check if the team has a bridge configured:
   - Use `bridge::discover()` to find the bridge dir
   - If found, load the bridge manifest

2. If the bridge type is "external" AND stdin is a terminal (interactive mode):
   - Use `cliclack::input` to prompt: "{display_name} bot token for {member_name} (optional, press Enter to skip)"
   - If the user provides a token (non-empty), store it via `LocalCredentialStore::new(state_path).store(member_name, token)`
   - If empty or non-interactive, skip silently

3. If token was not provided, print a note: "No bridge token provided. Add later with: bm bridge identity add {member_name}"

CRITICAL: The token goes to CredentialStore (system keyring via LocalCredentialStore), NOT to bridge-state.json, NOT to ralph.yml, NOT to config.yml. The CredentialStore also records the member name in bridge-state.json identities (metadata only, no token) so the member is known to the bridge system.
  </action>

  </verify>
  <done>Init wizard offers bridge selection (interactive and non-interactive). Hire prompts for optional token on external bridges. Bridge selection recorded in team botminter.yml. Token stored via CredentialStore.</done>
</task>
