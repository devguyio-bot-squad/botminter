1. **Add `keyring` dependency** to `crates/bm/Cargo.toml`:
   ```toml
   keyring = "3"
   ```
   The `keyring` crate provides cross-platform credential storage (macOS Keychain, Linux Secret Service/kernel keyring, Windows Credential Manager).

2. In `bridge.rs`, define the `CredentialStore` trait:
```rust
pub trait CredentialStore {
    fn store(&self, member_name: &str, token: &str) -> Result<()>;
    fn retrieve(&self, member_name: &str) -> Result<Option<String>>;
    fn remove(&self, member_name: &str) -> Result<()>;
    fn list(&self) -> Result<Vec<String>>;
}
```

3. Implement `InMemoryCredentialStore` for testing:
```rust
pub struct InMemoryCredentialStore {
    tokens: std::sync::Mutex<HashMap<String, String>>,
}
```
   This implements `CredentialStore` with an in-memory HashMap. Used by all unit and integration tests — avoids dependency on system keyring in CI. All behavior tests in this plan and downstream plans (02, 03) use `InMemoryCredentialStore`, NOT `LocalCredentialStore`.

4. Implement `LocalCredentialStore` backed by system keyring via the `keyring` crate. This is the local formation backend:
   - Constructor: `LocalCredentialStore::new(team_name: &str, bridge_name: &str, state_path: PathBuf)` — the team+bridge name forms the keyring service identifier (e.g., `botminter.{team}.{bridge}`).
   - `store(member, token)`: calls `keyring::Entry::new("botminter.{team}.{bridge}", member)?.set_password(token)?`. Also records the member name in bridge-state.json `identities` (username/user_id only, NO token) so `list()` can enumerate without scanning the keyring.
   - `retrieve(member)`: calls `keyring::Entry::new(...)?.get_password()`. Returns `Ok(None)` on `NoEntry` error.
   - `remove(member)`: calls `keyring::Entry::new(...)?.delete_credential()`. Removes from bridge-state.json identities too.
   - `list()`: reads member names from bridge-state.json `identities` keys (the keyring doesn't support enumeration, so bridge-state.json remains the index).
   - **Keyring error handling**: On `PlatformFailure` or other keyring backend errors (common on headless Linux, CI, containers), `store()` MUST return a clear error: `"System keyring not available. On Linux, install a Secret Service provider (e.g., gnome-keyring). For headless/CI environments, set BM_BRIDGE_TOKEN_{USERNAME} environment variables instead."` The `retrieve()` method should return `Ok(None)` with a warning on `PlatformFailure` (graceful degradation — env var override still works via `resolve_credential_from_store`).

5. **Migrate `BridgeIdentity` in bridge-state.json**: Make the `token` field optional with a default: `#[serde(default, skip_serializing)]`. This means:
   - Old bridge-state.json files with `"token": "..."` deserialize without error (token is read but ignored — never re-serialized)
   - New serializations never include the token field
   - `BridgeIdentity` keeps `username`, `user_id`, `created_at` as the identity record
   - Bridge-state.json remains the operational state file for service URLs, container IDs, rooms, and identity metadata — but secrets are no longer stored there
   - Add a behavior test verifying old files with token field deserialize correctly

5. Update `resolve_credential` to add a new function `resolve_credential_from_store`:
```rust
pub fn resolve_credential_from_store(
    member_name: &str,
    credential_store: &dyn CredentialStore,
) -> Result<Option<String>>
```
   This checks: (1) env var `BM_BRIDGE_TOKEN_{USERNAME}` first (uppercased, hyphens replaced with underscores via `env_var_suffix()` — e.g., `agent-alice` → `BM_BRIDGE_TOKEN_AGENT_ALICE`), (2) `credential_store.retrieve(member)` second. This is what `bm start` and `bm bridge` commands use.

   Also fix the existing `resolve_credential()` function to use the same `env_var_suffix()` helper for hyphen-to-underscore normalization (this is a pre-existing bug where `BM_BRIDGE_TOKEN_AGENT-ALICE` is an invalid env var name).
