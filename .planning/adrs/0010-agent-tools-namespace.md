# Separate `bm-agent` binary for agent-facing CLI

## Problem

BotMinter agents (brain process, coding agents inside Ralph loops) need CLI commands for inter-agent communication and coding-agent hook integration. Putting these under `bm agent` clutters `bm --help` with commands operators never invoke directly and creates confusion ("what does `bm agent` do?"). Agent commands have fundamentally different error handling requirements — a PostToolUse hook command must never block the coding agent, while an operator command should report errors clearly. Additionally, hook mechanisms are coding-agent-specific (Claude Code has PostToolUse hooks, Gemini CLI may have different mechanisms).

## Constraints

* Agent commands must not appear in `bm --help` — operators should never wonder what they do
* Hook commands must not cause the coding agent to treat a hook invocation as a failure
* The CLI must be coding-agent-aware — hooks are namespaced by coding agent
* Agent-agnostic commands (inbox) must be separate from coding-agent-specific commands (hooks)
* Both binaries must share domain modules — no code duplication

## Decision

Agent-facing commands live in a separate `bm-agent` binary, built as a second `[[bin]]` target in the same `crates/bm/` crate. This gives agents their own CLI with its own help, while keeping `bm --help` purely operator-facing.

```toml
# crates/bm/Cargo.toml
[[bin]]
name = "bm"
path = "src/main.rs"

[[bin]]
name = "bm-agent"
path = "src/agent_main.rs"
```

Both binaries share all domain modules (`brain::inbox`, etc.) via the crate's `lib.rs`.

### Command structure

```
# Agent-agnostic: brain-to-loop messaging
bm-agent inbox write <message> [--from <name>]
bm-agent inbox read [--format json|hook]
bm-agent inbox peek

# Coding-agent-specific: Claude Code hooks
bm-agent claude hook post-tool-use
```

### Hook error handling strategy

Hook commands always exit 0 and suppress stderr internally. This is the chosen strategy to satisfy the "must not cause failure" constraint. The tradeoff is loss of error visibility — hook errors are silently swallowed. If future debugging requires hook error observability, a structured log file (e.g., `.ralph/hook-errors.log`) can be added without changing the exit-code contract.

### Context resolution

`bm-agent` commands run inside a workspace with no interactive flags. They discover their context via walk-up directory search: starting from cwd, walk up looking for the `.botminter.workspace` marker file (same pattern as `git` finding `.git/`). This differs from `bm` commands which use `config::load()` + `--team` flags.

- `inbox` commands: error if workspace not found (the brain needs to know it's in a workspace)
- `hook` commands: silently exit 0 if workspace not found (the hook may fire outside a workspace context)

### Boundary with `bm` internal commands

The `bm` binary has hidden commands (`DaemonRun`, `BrainRun`) marked `#[command(hide = true)]`. These are **process entry points** — the `bm` binary re-invokes itself to spawn child processes. They are not agent-consumed tools.

`bm-agent` is a different category: commands consumed by agents during their work loop (messaging, hooks).

| Category | Binary | Examples |
|----------|--------|----------|
| Operator commands | `bm` | `bm start`, `bm hire`, `bm status` |
| Agent-consumed tools | `bm-agent` | `bm-agent inbox write`, `bm-agent claude hook post-tool-use` |
| Internal process entry points | `bm` (hidden) | `bm daemon-run`, `bm brain-run` |

### Domain module structure

Per ADR-0007, both binaries call into shared domain modules. The `bm-agent` command layer is thin:

- `agent_main.rs` — Clap parser and dispatch for `bm-agent`
- `brain::inbox` — inbox types, JSONL I/O, hook response formatting. Shared by both `inbox` and `claude hook` commands.

The `claude hook post-tool-use` command calls `brain::inbox::discover_workspace_root`, `brain::inbox::read_messages`, and `brain::inbox::format_hook_response` — the same domain functions as `inbox read --format hook`, but with silent error handling. No separate domain module needed for hooks in v1.

### Naming

`bm-agent` follows the convention of companion binaries (`cargo-fmt`, `git-credential-store`). The hyphen signals it's a companion to `bm`, not a standalone tool. The name is consistent with BotMinter terminology where "agents" are the coding agents and brain processes that do work.

### Stability contract

`bm-agent` subcommands follow the same alpha breaking-changes policy as `bm`. The `settings.json` hook configuration (`"command": "bm-agent claude hook post-tool-use"`) is regenerated by `bm teams sync` from the profile, so changes to command names propagate automatically.

## Rejected Alternatives

### Subcommand namespace (`bm agent inbox`, `bm agent claude hook`)

Rejected because: appears in `bm --help` alongside operator commands. Operators would see `agent` and wonder what it does. The subcommand approach was tried and found confusing.

### Flat top-level commands (`bm inbox`, `bm hook`)

Rejected because: pollutes the operator-facing CLI with commands operators don't use directly. As agent capabilities grow, this creates a confusing mix.

### Shell script shims for hooks

Rejected because: requires creating, chmod-ing, extracting, and surfacing shell scripts during workspace provisioning. A Rust binary handles error suppression, is testable, and eliminates an entire artifact class.

### Hidden commands (`#[command(hide = true)]`)

Rejected because: hiding commands obscures the agent contract. A separate binary makes the boundary explicit and discoverable via `bm-agent --help`.

### Separate crate (`crates/bm-agent/`)

Rejected because: adds boilerplate (separate `Cargo.toml`, dependency management) for no benefit. A second `[[bin]]` in the same crate shares all source and builds in one `cargo build`.

### Coding-agent-unaware hook namespace (`bm-agent hook post-tool-use`)

Rejected because: PostToolUse is a Claude Code concept. Namespacing by coding agent (`claude hook`, `gemini hook`) prevents collision and makes the agent boundary explicit.

### Alternative names (`bm-internal`, `bm-machine`)

- `bm-internal` — suggests hidden/private; agent commands are explicitly public and documented
- `bm-machine` — unclear and unfamiliar

## Consequences

* `bm --help` is 100% operator-facing — no agent commands visible
* `bm-agent --help` documents the full agent contract
* Both binaries are built from one `cargo build -p bm` — no extra build steps
* `cargo install bm` installs both binaries
* Claude Code `settings.json` references `bm-agent claude hook post-tool-use` directly
* Adding a new coding agent's hooks is additive (new subcommand group under `bm-agent`)
* Hook errors are silently swallowed — debugging requires explicit logging in a future iteration
* The `bm-agent` binary adds ~0 to binary size (same code, different entry point)

## Anti-patterns

* **Do NOT** add agent-consumed commands to the `bm` binary — they belong in `bm-agent`. Operators should never see agent commands in their help output.
* **Do NOT** create shell script shims for coding agent hooks — use `bm-agent <agent> hook <event>` commands that handle error suppression in Rust.
* **Do NOT** make hook commands exit non-zero on any condition — hooks must always exit 0.
* **Do NOT** share error handling patterns between `inbox` and `hook` commands — inbox commands report errors normally (exit 1, stderr), while hook commands always succeed (exit 0, no stderr).
* **Do NOT** put coding-agent-specific concepts under agent-agnostic namespaces — PostToolUse belongs under `claude hook`, not a generic `hook` namespace.
* **Do NOT** use `bm-agent` for internal process entry points — `DaemonRun`/`BrainRun` remain hidden `bm` commands.
